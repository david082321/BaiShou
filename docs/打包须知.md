# 白守 (BaiShou) 项目打包与部署规范

本文档用于规范“白守”客户端的正规化构建与密钥分发流程。本项目针对 Android 签名采用了 **无实体文件 (Fileless) 的 Base64 凭证注入方案**，以实现本地开发与 CI/CD 管道的双重脱敏隔离。

## 1. 架构方案概述

为防止证书泄露并提升流水线兼容性，项目已彻底废弃本地物理 `.jks` 证书。

- **本地构建环境**：通过注入 `android/key.properties` 的 `storeBase64` 变量，在 Gradle `assembleRelease` 任务触发时，于堆内存中即时解码并生成临时凭证文件至 `build/app/tmp_keystore.jks`，任务结束后伴随 `clean` 指令销毁。
- **CI/CD 构建环境**：通过 GitHub Actions Secrets 管理环境变量，在流水线初始化阶段解密至 Runners 临时目录。

---

## 2. 本地构建前置配置 (Local Environment)

### 2.1 初始化 `key.properties`

在首次克隆项目或迁移开发机时，需在 `android/` 目录下手动创建 `key.properties` 文件。

> **注意**：该文件已被加入 `.gitignore`，严禁通过任何手段（含 `--force`）推送到远端仓库。

### 2.2 配置文件模板

```properties
# android/key.properties
storePassword=(Store密文)
keyPassword=(Key密文)
keyAlias=upload
storeBase64=(JKS实体的完整Base64编码字符串)
```

### 2.3 执行构建命令

配置就绪后，于项目根目录执行 Flutter Release 指令：

```bash
flutter build apk --release --no-tree-shake-icons
```

_注：由于项目中包含了特殊字体集，`--no-tree-shake-icons` 参数为强制要求，否则会导致 Resource Shrinker 执行崩溃。_

## 3. GitHub CI/CD 云端构建配置 (Remote Environment)

项目的持续集成管线配置于 `.github/workflows/release.yml`。流水线监听 `v*` 规则的 Tag push 事件，并自动拉取 Secrets 执行跨平台矩阵构建。

### 3.1 必须配置的环境变量映射

需在仓库控制器中（`Settings -> Secrets and variables -> Actions -> Repository secrets`）全量注入以下生产级变量：

| Secret 键名 (Key)         | 对应值说明 (Value)                         | 安全层级        |
| :------------------------ | :----------------------------------------- | :-------------- |
| `ANDROID_KEYSTORE_BASE64` | `upload-keystore.jks` 的 Base64 安全编码串 | 极高 (Critical) |
| `ANDROID_KEY_ALIAS`       | 证书签发别名 (通常为 `upload`)             | 中等 (Medium)   |
| `ANDROID_KEY_PASSWORD`    | Key 实例鉴权密文                           | 高 (High)       |
| `ANDROID_STORE_PASSWORD`  | Keystore 容器鉴权密文                      | 高 (High)       |

### 3.2 触发云部署流程

当功能迭代分支合并至 `main` 后，执行以下标准 Git 流触发完整 Release：

```bash
git tag vX.Y.Z
git push origin vX.Y.Z
```

Actions Runner 将在容器内拉取上述配置，并自动向 GitHub Releases 面板交付带官方签名的 `.apk` 安装包以及桌面端构建产物。
