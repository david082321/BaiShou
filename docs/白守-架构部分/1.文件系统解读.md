# 白守 (BaiShou) 文件系统与同步流架构解读

> 本文档详细阐述了白守 App 核心的“本地 Markdown 物理层 + SQLite 索引层 + UI 内存层”的三层架构设计，以及如何通过精心设计的单向数据流解决多端或外部应用修改带来的并发冲突难题。

## 1. 核心设计原则

白守采用的是 **“以本地 Markdown 文件为单一事实来源（Single Source of Truth），以 SQLite 索引表为检索加速引擎”** 的混合架构。

- **为何避免纯数据库存储？**
  日记属于用户的高价值数字资产，强绑定应用内数据库容易导致长期的数据孤立。Markdown 文本兼具人类可读性与跨平台可移植性（可无缝接入 Obsidian、VS Code 等），契合“白守”作为数据托管者的设计初衷。
- **为何需要 SQLite 索引图谱？**
  若每次启动或全文搜索（FTS）均对成百上千个本地文本文件进行全量 I/O 读取，移动端和桌面应用将面临严重的性能瓶颈。

因此，架构确立了 **“物理层负责持久化存储，SQLite 缓存层负责结构化与检索，内存池负责视图渲染”** 的基础逻辑。

---

## 2. 架构拓扑：三层单向数据流 (Unidirectional Flow)

为了解决外部编辑器（如 Obsidian）后台修改文件、同步盘（OneDrive/iCloud）后台拉取文件时的冲突，以及 App 自身保存时产生的 Watcher 冗余事件，我们将同步流严格分为三个独立解耦的调度层级：

### 2.1 事件捕获与防抖网关 (`FileStateScheduler`)

**定位**：系统级 OS File Watcher。它是文件系统变动事件的唯一入口。

- **事件防抖 (Event Debouncing)**：操作系统在处理文件保存操作时往往会产生冗余事件（特别是 Windows 下的安全保存可能同时触发 `create`、`modify`、`move` 等操作）。调度层通过 `rxdart` 对同一路径的事件流应用 `debounceTime(200ms)`，将短时间内的事件风暴规约为单一的有效变更脉冲。
- **并发状态锁 (Write Suppression)**：应用内部发起的日记保存操作同样会触发到底层 Watcher。为避免事件死循环，在执行磁盘写操作前，系统会向调度器注册一条路径并发锁 `suppressPath(2s)`。调度器在捕获变动事件时，若发现对应路径处于锁定期，将主动丢弃该事件。

**输出**：一条经过清洗去重、无自我触发环路的纯净文件变动流 (`cleanFileEvents`)。

### 2.2 数据校验与一致性同步层 (`ShadowIndexSyncService`)

**定位**：数据结构化与 SQLite 状态同步枢纽。

- **哈希校验机制 (Hash Verification)**：该层监听来自网关的数据流。在执行昂贵的数据库 I/O 乃至触发 UI 更新前，服务会首先计算目标文件的 **MD5 Hash**，并与当前 SQLite 内存储的 Hash 进行严格比对。
  - 若 Hash 一致：判定为文件系统造成的冗余触发（如元数据变更），直接丢弃该任务。
  - 若 Hash 变更（或触发新增/删除）：异步解析 Markdown 内容，并精确更新或删除 SQLite 影子条目（同步维护 FTS5 虚拟视图）。
- **状态载荷封装 (Payload Encapsulation)**：数据结构化完成后，服务避免仅发送简单的“更新通知”，而是将解析后的元数据信息（Preview, Tags, Date 等）与状态标识符统一封装在一个 `JournalSyncEvent` 数据载荷（Payload）中。

**输出**：一个全状态、自包含的同步结果分发流 (`syncEvents`)。

### 2.3 内存映射与视图层 (`VaultIndexNotifier` & `DiaryRepository`)

**定位**：高响应的内存管理池与 UI 状态容器。

- **规避重复查询**：监听组件直接接收包含详尽元数据信息的 `JournalSyncEvent` 载荷包。业务层无需再次向底层 SQLite 发起查询校验，而是直接利用提供的 `DiaryMeta` 和变动标志位 (`isChanged`)，在内存状态列表中执行时间复杂度为 $O(n)$ 的 `upsert` 或 `remove` 操作。
- **高可用响应**：通过消除二次数据库查询延迟，该机制确保了当外部文件系统状态发生变更后，UI 视图能以近乎零延迟的效率完成重绘重排。

## 3. 面向外部环境的鲁棒性兼容

- **安全保存 (Safe-Save) 兼容机制**：诸如 Obsidian 等现代编辑器在操作文件时，倾向于新建临时文件（如 `.sync-conflict`）并在写入完成后覆盖（`move`）原文件，以防止数据损坏。`FileStateScheduler` 专门增设了对 `FileSystemMoveEvent.destination` 属性的捕获解析，确保外部编辑器的重命名覆盖操作能够被正确映射到文件变更流中。
- **异步锁避让策略**：Windows 操作系统在执行文件保存瞬间可能会挂起文件读取锁，`ShadowIndexSyncService` 采取了缓冲读取策略，在接收到变动调度后主动延迟短暂时钟周期再进行 I/O 操作，显著降低了哈希校验异常率。

## 4. 数据更新流程 (Internal Update Workflow)

在使用白守内置编辑器进行内容的写操作（新增、编辑或删除）时，系统遵循以下执行路径以确保数据的一致性并主动规避事件冲突：

1. **注册变更抑制 (Write Suppression)**：
   在执行任何磁盘写入操作之前，`DiaryRepository` 会向 `FileStateScheduler` 注册目标路径的写入标记。此操作将使调度器在后续的 2 秒窗口内主动忽略该路径触发的任何 Watcher 事件，有效拦截了由于自身写入操作引起的“回声”干扰。
2. **物理数据持久化 (Physical Persistence)**：
   仓库层通过 `JournalFileService` 将 Markdown 内容异步持久化至本地磁盘。
3. **影子索引同步 (Index Synchronization)**：
   同步调用 `ShadowIndexSyncService.syncJournal`。该步骤会重新读取物理文件，计算最新的 MD5 Hash，并原子化地更新 SQLite 影子索引（包含全文本搜索索引 FTS5），确保数据库层与物理文件层状态绝对同步。
4. **内存状态响应 (Memory View Upsert)**：
   最后，系统通过载荷分发流直接更新 `VaultIndexNotifier` 中的内存状态。由于第 1 步中的抑制逻辑，Watcher 在文件落地后即便被触发，也无法干预当前已处于最新稳态的内存数据，从而实现了无抖动的线性一致性。

## 5. 故障恢复与一致性保障 (Consistency & Recovery)

在涉及到物理文件系统与关系型数据库这两种异构介质的并发操作时，白守通过以下容错设计确保系统的最终一致性：

### 5.1 真理优先持久化 (Truth-First Persistence)

系统遵循严格的线性顺序：**物理 Markdown 写入 -> SQLite 影子索引同步 -> 内存视图载荷分发**。

- 若物理写入阶段失败（如磁盘空间不足、IO 异常），程序将立即抛出异常并中断后续所有逻辑。此时，应用状态与 UI 将维持在修改前的状态，确保了“索引绝不会领先于物理真理”。
- 物理层“写完即成功”的设计，将复杂的介质同步问题简化为了“以文件系统为准”。

### 5.2 哈希自愈机制 (Self-Healing via MD5)

若发生物理写入成功但数据库事务异常的极端情况（如应用闪退），系统依赖 **MD5 哈希状态锚点** 实现最终一致性：

- **状态失配检测**：在下一次应用启动、后台全量扫描或任何触发该路径变动的时刻，`ShadowIndexSyncService` 会对比物理文件与数据库中存储的哈希快照。
- **强制同步**：只要哈希不一致，同步层会判定数据库条目已过期，并立即启动原子同步任务进行修正，从而实现数据的自解析与状态自愈。

### 5.3 内存优先的视图隔离

由于在更新流程（第 4 章）的末尾采用了直接的内存载荷分发机制，即便数据库同步层出现短时物理延迟，用户在当前会话内看到的呈现依然是基于最新解析结果的。这种设计有效隔离了底层磁盘 I/O 波动对用户感知层面的影响。

---

## 6. 总结：系统设计的演进

在系统迭代初期，曾依赖“1秒高频轮询器”作为 Watcher 不稳定状态的兜底修复方案。
借由明确确立的 **Scheduler -> SyncService -> VaultIndex** 单向数据流范式，系统将底层 Watcher 的容错性及抗干扰能力重塑至稳态，从而使废除具有高昂 I/O 开销的脏检查轮询引擎成为可能。

至此，白守存储架构完成闭环：在开放物理 Markdown 文件访问权限的同时，通过精密的状态机和事件流管理，提供了媲美封闭云原生数据库般的系统流畅性与数据一致性。
